# Browser Toolset Recommendations

## Summary

Our browser toolset is a solid minimal implementation, but `browser-use/browser-use` is more mature in three areas:

- Better browser state: richer, compressed, more reliable DOM observations.
- Better action loop: structured memory/evaluation/planning and multi-action execution.
- Better reliability: page-stability waits, dialog/download/watchdog handling, stale element recovery, and loop detection.

The highest ROI improvements are not simply adding more tools. The biggest wins are improving `browser_snapshot`, tightening tool semantics, and adopting upstream's prompt/action-loop discipline.

## Local Implementation Reviewed

Relevant files:

- `packages/server/src/tools/toolsets/browser.ts`
- `packages/server/src/lib/browser/browser-manager.ts`
- `packages/server/src/lib/browser/instructions/browser.md`
- `packages/shared/src/browser/electron.ts`
- `apps/desktop/src/main/browser-manager.ts`

Current agent-facing tools:

- `browser_snapshot`
- `browser_navigate`
- `browser_interact`
- `browser_wait`
- `browser_screenshot`
- `browser_dialog`
- `browser_content`
- `browser_batch`

The design is pragmatic: one grouped toolset, a WebSocket bridge to Electron, serialized browser commands, refs from snapshots, and batching up to 5 actions.

## Upstream Browser-Use Comparison

Upstream has similar action categories but deeper infrastructure:

- Actions: navigate/search/back, click, input/type, keyboard, scroll, tabs, select dropdown, screenshot, extract, search page, find elements, upload file, file operations, and done.
- DOM state: CDP-based DOM snapshot, accessibility tree, layout metrics, visibility, paint order, iframe/shadow DOM handling, scroll containers, stable numeric indexes.
- Prompt loop: explicit `thinking`, `evaluation_previous_goal`, `memory`, `next_goal`, optional plan updates, and multi-action output.
- Runtime: event bus, watchdogs for DOM, popups, downloads, storage state, crashes, security, and tab lifecycle.
- Efficiency: multi-action sequences, page-change interruption, cheap `search_page` / `find_elements`, expensive extraction only when needed.

## Major Gaps

### 1. `browser_snapshot` Is Too Shallow

Our snapshot is generated by a custom injected DOM walker in `apps/desktop/src/main/browser-manager.ts`.

Current limitations:

- Uses simple DOM traversal, not browser accessibility/CDP snapshot.
- Does not handle iframes.
- Skips SVG entirely.
- Does not handle shadow DOM.
- Interactivity detection is basic: tag, role, `onclick`, `tabindex`, cursor pointer.
- Visibility detection only checks dimensions, `visibility`, and `display`.
- Does not detect occlusion, viewport position, paint order, disabled ancestors, inert elements, overlays, or hidden-by-parent edge cases.
- Refs are rebuilt each snapshot as `e1`, `e2`, etc., so refs are not stable across steps.
- Snapshot can include too much irrelevant text and miss important interactive elements.

Recommendation:

- Replace or augment `SNAPSHOT_SCRIPT` with a richer snapshot model.
- Include viewport bounds and element bounding boxes.
- Include iframe and shadow DOM traversal where possible.
- Mark scrollable containers.
- Add stable element identity where possible.
- Mark newly appeared elements since last snapshot.
- Include only visible/interactable elements plus useful nearby text.
- Add `disabled`, `checked`, `selected`, `expanded`, `required`, `value`, `placeholder`, and `aria-*` attributes.

### 2. Several Tool Options Are Advertised But Not Implemented

The schemas expose options that the Electron executor ignores.

Examples:

- `click.doubleClick`, `button`, `modifiers`, and `timeoutMs` are accepted, but desktop click just runs `.click()`.
- `type.slowly` is accepted but ignored.
- `screenshot.ref`, `fullPage`, `webp`, and element screenshots are accepted, but `capturePage()` always captures the viewport and only returns PNG/JPEG.
- `resize` returns `Resize requested...` but does not resize anything.
- `dialogState` always returns `{ open: false }`.
- `handleDialog` always says no handling is required.
- `timeoutMs` is passed through many actions but mostly unused by desktop execution.
- `browser_content.extract` accepts `query`, but the implementation ignores the query and returns `innerText` for a selector/body.

Recommendation:

- Either implement these options or remove/de-emphasize them from schemas/descriptions.
- Highest priority to implement: real click, real type, real dialog tracking, real full-page/element screenshot, and real resize.

### 3. Direct DOM Mutation Makes Interactions Less Realistic

Current click/type/select are mostly JavaScript DOM operations:

- Click: `document.querySelector(...).click()`
- Type: assign `el.value`, dispatch `input`/`change`
- Select: set `option.selected`, dispatch events

This is fast, but less reliable on modern apps because React/Vue/Svelte/custom components often depend on trusted pointer/keyboard events, focus behavior, composition events, `keydown`/`keyup`, pointer down/up, blur, and framework-specific event ordering.

Recommendation:

- For click, compute element rect from ref and use Electron `sendInputEvent` mouse down/up at the center.
- For type, focus element, optionally clear with Ctrl+A/Backspace, then use keyboard insertion or `insertText`.
- Keep JS fallback only when input events fail.

### 4. Prompt Is Good But Too Thin

Our prompt already has good principles:

- Snapshot first.
- Use latest refs.
- Batch short sequences.
- Use content tools for fast checks.
- Verify before completion.

Upstream adds important missing behavior:

- Explicit previous-step evaluation.
- Memory to avoid loops.
- Next immediate goal.
- Strong data-grounding rules.
- Step-budget awareness.
- Filter-first behavior for shopping/search tasks.
- Popup/modal/cookie handling priority.
- Autocomplete-specific workflow.
- Do not retry same failure 2-3 times.
- Page-changing actions must be last.
- If batch is interrupted, continue remaining intent next step.

Recommendation:

- Add a compact agent-loop section to `browser.md`.
- Require the agent to evaluate the previous browser action as success/failure/uncertain.
- Require tracking what has been tried.
- If the same action fails twice, require a different strategy.
- Prefer `browser_content.search_page` before scrolling for specific text.
- Prefer filters/sort controls before browsing result lists.
- Handle overlays before the main task.
- Do not report data unless observed in snapshot/content/screenshot.

### 5. `browser_batch` Is Useful But Less Powerful Than Upstream Multi-Action Output

Our `browser_batch` is a good efficiency primitive. It stops on known page-changing actions or detected state changes.

Weaknesses:

- Page state detection uses active tab URL/title only.
- DOM changes without URL/title changes may not interrupt.
- `click` is not treated as sequence-terminating, even though it often changes pages.
- Batch results include nested output objects, which may be verbose.
- It cannot reference newly created refs inside the same batch.

Recommendation:

- Before/after state should include a lightweight DOM revision fingerprint: URL, title, `document.readyState`, `location.href`, focused element, visible interactive element count/hash, and possibly a body text hash slice.
- Treat `click` and `press Enter` as potentially page-changing and stop if DOM/URL changes.
- Return a concise batch summary plus only the important output, not all nested objects unless requested.

### 6. Missing File Upload

Upstream supports file upload. Our command schema has no upload action.

Recommendation:

- Add `browser_interact` action `upload`.
- It should take `ref` and a file path from an allowed workspace/download list.
- Desktop side should call Electron/DOM file input APIs safely.

### 7. Dialog And Popup Handling Is Currently Mostly Fictional

The toolset advertises dialog handling, but implementation returns no dialog.

Recommendation:

- Track Electron JavaScript dialogs using relevant `WebContents` events.
- Store last dialog type/message.
- Implement accept/dismiss/prompt.
- Include recently dismissed dialog messages in `browser_snapshot`.

### 8. Downloads Are Tracked In UI State But Not Agent-Friendly

Desktop tracks downloads and browser state includes downloads, but tools do not expose a clear download workflow.

Recommendation:

- Add `browser_content` or `browser_navigate` action `downloads`.
- Snapshot should include recent downloads.
- Add prompt guidance: if PDF/download appears, verify path before claiming success.

### 9. Tabs Are Session-Simulated, Not Real Multi-Tab Browser Targets

Our tabs are a Map of URLs loaded into one webview. Switching tabs reloads the active webview URL. This is simpler but loses real tab state such as scroll, DOM state, form state, history, JS runtime state, and pending auth flows.

Recommendation:

- Short term: document limitation in prompt/tool descriptions.
- Medium term: persist per-tab scroll and maybe form state.
- Long term: use actual BrowserViews/WebContents per tab if agents need robust multi-tab workflows.

### 10. Navigation Waits Are Basic

`navigate` calls `browser.loadURL()` and returns. There is no clear page-stability policy beyond Electron's load behavior.

Recommendation:

- After navigation, wait for `document.readyState !== "loading"`.
- Add a short network/DOM quiet period.
- If snapshot is empty on HTTP(S), wait/reload once.
- Respect `timeoutMs`.

## Prioritized Roadmap

1. Fix schema/implementation mismatches.
2. Improve real interactions.
3. Upgrade snapshot.
4. Strengthen prompt.
5. Improve page-change detection in `browser_batch`.
6. Implement dialogs/downloads/file upload.
7. Consider stable numeric refs or stable ref identity.

## Concrete Prompt Changes To Borrow

Add these concepts to `packages/server/src/lib/browser/instructions/browser.md`:

- Always evaluate whether the last browser action succeeded before proceeding.
- Never repeat the same failed action more than twice.
- Use `browser_content.search_page` for specific text before scrolling.
- Use `browser_content.find_elements` to inspect repeated DOM structures.
- Treat extraction as expensive; only use it when snapshot/search cannot answer.
- Handle popups, modals, cookie banners, and overlays before main actions.
- For autocomplete fields: type, snapshot/wait, then choose the correct new suggestion.
- For result-list tasks with filters, apply filters/sort before browsing results.
- Before final answer, verify every requested fact/action against observed browser output.
- If blocked by login, 403, bot detection, CAPTCHA, or rate limits, do not loop; report or try an alternative route.

## Best Immediate Engineering Wins

If we only do three things:

1. Make click/type real browser input events instead of DOM mutation.
2. Make snapshot richer and less noisy.
3. Remove or implement fake options like dialog, full-page screenshot, element screenshot, resize, doubleClick, slowly, and extract query semantics.

These should make the agent noticeably better, faster, and more reliable without a large architecture rewrite.
